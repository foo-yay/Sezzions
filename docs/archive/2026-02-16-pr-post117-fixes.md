## Summary
This PR brings post-Issue-117 follow-up fixes back into `main`.

- Exclude soft-deleted redemptions from realized/basis rebuild paths.
- Improve redemption partial/full confirmation so “full cashout” is based on expected TOTAL balance (not just redeemable).
- Ignore soft-deleted redemptions when computing the session delete “DELETION IMPACT” warning.

## Why
After PR #119 (Issue #117) merged, a few UX/data-integrity edge cases were found during real usage:
- Deleted redemptions still appeared to consume basis / show in Realized.
- Redeeming all *redeemable* was being treated as “full cashout” even when total balance remained.
- Session delete warnings were still counting redemptions that had been soft-deleted.

## Changes
- `services/recalculation_service.py`: exclude `deleted_at` redemptions from rebuild queries.
- `repositories/realized_transaction_repository.py`: exclude realized rows whose redemption is soft-deleted.
- `ui/tabs/redemptions_tab.py`: confirmation classification now compares against expected TOTAL balance; adds unit-tested classifier.
- `services/game_session_service.py`: session delete impact ignores soft-deleted redemptions.

## Tests
- `pytest -q` (882 passed, 1 skipped)
- New regression tests:
  - `tests/unit/test_redemption_confirmation_classification.py`
  - `tests/unit/test_game_session_deletion_impact.py`

## Manual check (≤5 min)
- Delete a redemption that occurs after a session, then delete the session: ensure no “future redemption(s)” warning.
- Edit/create a redemption where redeemable is fully redeemed but total balance remains: ensure it is treated as partial (or at least not warned as full cashout).

## Pitfalls / Follow-ups
- Cashflow FIFO “basis recovery” vs session-driven “taxable P/L” can be confusing when a purchase is made after profit is earned but before redemption.
  - Follow-up idea: add an optional, separate metric/column that explains profit-withdrawal vs basis-recovery without changing FIFO semantics.
