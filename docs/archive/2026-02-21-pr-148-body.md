## Summary

Implements Issue #148: Redemption Cancel / Uncancel.

Closes #148

## Changes

### Schema
- Added `status` (PENDING | CANCELED | PENDING_CANCEL), `canceled_at` (UTC), `cancel_reason` columns to `redemptions` via migration.

### Core Accounting
- **Two-event delta model** in `compute_expected_balances`: a CANCELED redemption contributes a debit at `redemption_date` and a credit at `canceled_at`, preserving balance continuity across the full timeline.
- **RecalculationService**: both FIFO rebuild queries now exclude CANCELED/PENDING_CANCEL rows (`COALESCE(status,'PENDING') NOT IN ('CANCELED','PENDING_CANCEL')`).
- **process_pending_cancels** moved outside the `if recalculate_pl:` guard so it fires whenever a session transitions Active to Closed, regardless of the recalculate_pl flag.

### Services
- `RedemptionService.cancel_redemption`: validates PENDING status, reverses FIFO, sets CANCELED (or PENDING_CANCEL if active session).
- `RedemptionService.uncancel_redemption`: validates CANCELED status, re-applies FIFO.
- `RedemptionService.process_pending_cancels`: batch-executes all PENDING_CANCEL for a user/site pair when session closes.

### AppFacade
- `cancel_redemption(redemption_id, reason)`: checks for active session, delegates to service, triggers rebuild.
- `uncancel_redemption(redemption_id)`: delegates to service, triggers rebuild.
- `GameSessionService` now receives `redemption_service` at construction.

### Notification / Bulk updates
- `NotificationRulesService.evaluate_redemption_pending_rules`: guards against `settings=None`; SQL filter for `status='PENDING'` already in place.
- `bulk_update_redemption_metadata`: skips CANCELED rows.

### UI (redemptions_tab.py)
- Cancel toolbar button (PENDING selection), Uncancel toolbar button (CANCELED selection).
- Cancel dialog: warns about active-session deferral, accepts reason text.
- CANCELED rows rendered gray (#95a5a6), PENDING_CANCEL rows purple (#8e44ad).
- "Pending" quick filter excludes CANCELED and PENDING_CANCEL rows.

## Pitfalls / Follow-ups

- PENDING_CANCEL rows retain their FIFO allocations until the session closes. If a user deletes an active session without closing it, PENDING_CANCEL rows will remain in that state. A follow-up issue should handle orphaned PENDING_CANCEL cleanup.
- The `NotificationRepository` stores to `settings.json` (not SQLite), making E5-style integration tests fragile in-process (cooldown state bleeds between tests). A future issue should migrate notifications to SQLite or add per-test isolation.
