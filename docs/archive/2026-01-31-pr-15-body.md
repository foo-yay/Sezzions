## Changes

### New Files

#### `ui/base_table_model.py` (125 lines)
- `ColumnDefinition`: Metadata class for table columns (label, key, formatter, alignment, width_hint)
- `BaseTableModel(QAbstractTableModel)`: Shared base class for grid tabs
  - Object-oriented: rows are domain objects, attribute-based access
  - `set_data(data)`: Replace model data with `beginResetModel`/`endResetModel`
  - `get_row_object(row)`: Retrieve underlying object for a row
  - `data(index, role)`: Handles DisplayRole, TextAlignmentRole, UserRole
  - `flags(index)`: Returns `ItemIsEnabled | ItemIsSelectable` (no editing support)
  - Subclasses override `data()` for custom formatting/coloring

## Architecture Decision

**Issue #15 Scope Reduced:**
- Originally included inline editing (Phase 2/3)
- Inline editing removed after strategic decision: keep edits in dialogs
- QTableView migration is now optional improvement, not prerequisite for editing

**Migration Status:**
- âś… Migrated: Sites (pilot successful)
- âŹł Remaining Setup tabs (6): Users, Cards, Redemption Method Types, Redemption Methods, Game Types, Games
- âŹł Remaining non-Setup tabs (5): Purchases, Redemptions, Game Sessions, Expenses, Unrealized

**Benefits of QTableView Architecture:**
- Cleaner separation: presentation (view) vs data (model)
- Built-in proxy model support for sorting/filtering without re-populating
- More idiomatic Qt architecture for data-driven UIs
- Foundation for future advanced features (custom delegates, lazy loading)
- No manual row-by-row population loops

**Trade-offs:**
- Requires rewriting selection/export logic to use model/proxy
- TableHeaderFilter (column filtering UI) not compatible with QTableView yet
- More boilerplate for simple tabs

## Pitfalls / Follow-ups

**Known Limitations:**
- Column filtering UI (TableHeaderFilter) not yet implemented for QTableView
