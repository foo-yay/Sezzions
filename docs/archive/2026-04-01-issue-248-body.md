## Problem

UsersTab, SitesTab, and CardsTab each contain ~1200 lines with ~73% structural duplication. The shared skeleton (state management, keyboard navigation, column resize, selection, pagination, search, copy-to-clipboard, context menus, sort/filter, infinite scroll, CRUD submit/delete) is copy-pasted across all three files with only variable names changed.

This means:
- Every behavioral fix (e.g., Ctrl+F select-all) must be applied to N files.
- New entity tabs require cloning an entire 1200-line file.
- Bugs fixed in one tab may not be fixed in others.

This violates PROJECT_SPEC.md Design Principles 1, 3, 6, and 7.

Additionally, 3 shared components live in the wrong folder:
- `ExportModal.jsx` is in `UsersTab/` but used by all tabs.
- `UserHeaderFilterMenu.jsx` is in `UsersTab/` with entity-prefixed name but used by all tabs.
- `TableContextMenu.jsx` is in `UsersTab/` but used by all tabs.

## Proposal

Extract shared entity table infrastructure into reusable hooks and components:

### Phase 1: Shared hook (`useEntityTable`)
A single custom hook that owns:
- Core data state (items, status, search, filters, sort)
- Selection state (selected IDs, anchor, column keys, cells)
- Loading/pagination state
- Keyboard handlers (Cmd+C, Cmd+A, Cmd+F, Escape, Arrow nav, Shift+Click)
- Column resize logic
- Copy-to-clipboard
- Sort/filter toggle functions
- Data loading (paginated fetch with fallback)
- CRUD submit/delete flows
- Context menu state

Configured via a config object:
```js
useEntityTable({
  entityName: "cards",
  apiEndpoint: "/v1/workspace/cards",
  columns: cardTableColumns,
  initialForm: initialCardForm,
  initialColumnFilters: initialCardColumnFilters,
  pageSize: 100,
  responseKey: "cards",        // key in API response payload
  getCellDisplayValue,         // entity-specific formatter
  normalizeForm,               // entity-specific form normalizer
  extraLoadFns,                // e.g., Cards loads users list too
})
```

### Phase 2: Shared component (`EntityTable`)
A generic table component that renders:
- Toolbar with breadcrumb
- Configurable action buttons (Add, View, Edit, Delete, Export, Refresh)
- **Custom action button slots** for entity-specific actions (e.g., "Mark Received", "Mark Processed" on Redemptions)
- Search bar with clear
- Table header with sort indicators + filter menus
- Table body with row/cell selection, highlight, resize
- Status bar
- Back-to-top button
- Context menu portal

### Phase 3: Relocate and rename misplaced shared components
- Move `ExportModal.jsx` to `common/ExportModal.jsx`
- Move `UserHeaderFilterMenu.jsx` to `common/TableHeaderFilterMenu.jsx` (rename)
- Move `TableContextMenu.jsx` to `common/TableContextMenu.jsx`

### Phase 4: Migrate existing tabs
Each entity tab shrinks to ~150-300 lines containing only:
- Column definitions
- Form fields and validation
- Cell display value formatter
- Modal component (entity-specific form layout)
- Any custom action buttons and their handlers

### Extension points (for future tabs like Redemptions, Purchases)
- **Custom toolbar actions**: Tabs can inject extra buttons (with visibility predicates based on selection state). E.g., Redemptions adds "Mark Received" visible only when pending items are selected.
- **Custom context menu items**: Entity-specific right-click actions.
- **Pre-load hooks**: Some tabs need to load auxiliary data before rendering (e.g., Cards loads users for the FK dropdown).
- **Custom cell renderers**: Override default text rendering for specific columns (e.g., status chips, formatted percentages).

## Scope

- IN: useEntityTable hook, EntityTable component, component relocation, migration of UsersTab/SitesTab/CardsTab
- IN: Custom action button extension point design
- IN: Shared utility extraction (isTextEntryElement, parseNumericValue, computeCellStats, downloadCsv)
- OUT: New entity tabs (Redemptions, Purchases, etc.)
- OUT: Backend changes
