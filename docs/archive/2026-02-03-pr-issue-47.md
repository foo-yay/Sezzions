## Summary
Fixes Issue #47: editing a purchase no longer self-includes the purchase when computing the expected **pre-purchase** SC balance.

## What changed
- Standardizes purchase balance-check cutoff to “1 second before purchase” (date+time safe; handles midnight rollover).
- Applies this consistently in:
  - PurchasesTab edit flow (warning dialog)
  - PurchaseDialog live balance-check label
- Adds a headless regression test reproducing the Zula example (20.75 vs 30.75 mismatch).

## Validation
- `pytest -q` (610 tests) ✅

## Pitfalls / follow-ups
- The current UI approach uses a 1-second cutoff, which can be ambiguous if multiple purchases share the exact same timestamp. A future enhancement could pass an `exclude_purchase_id` into expected-balance computation to avoid relying on time epsilon.
