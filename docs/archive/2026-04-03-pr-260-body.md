## Summary

Phase 3a of the web port plan: full-stack Purchases CRUD entity. Purchases are the first transaction entity and create FIFO cost-basis lots.

Closes #260

## Changes

### Backend
- **HostedPurchase model** (`services/hosted/models.py`): Decimal validation for amount (> 0), required fields (user, site, date), sc_received defaults to amount, remaining_amount defaults to amount
- **HostedPurchaseRepository** (`repositories/hosted_purchase_repository.py`): Full CRUD with JOIN on users, sites, cards for display names. Ordered by date DESC for transaction listing
- **HostedWorkspacePurchaseService** (`services/hosted/workspace_purchase_service.py`): Workspace-scoped CRUD delegation
- **API endpoints** (`api/app.py`): 5 endpoints at `/v1/workspace/purchases` (GET list, POST create, PATCH update, DELETE single, POST batch-delete)

### Frontend
- **PurchasesTab** (`web/src/components/PurchasesTab/`): EntityTable config with extraLoaders for users, sites, cards
- **PurchaseModal**: View/edit/create modes with TypeaheadSelect for User, Site, Card (filtered by user). Currency formatting for display
- **Activity nav group**: New sidebar section in AppShell alongside Setup, with `/activity/:tabKey` routing
- **Purchases icon**: Dollar sign SVG icon in Icon component

### Tests
- 22 new tests: model validation (6), service CRUD happy paths (7), edge cases (4), failure injection (4), model defaults (1)
- Full suite: 1271 passed, 1 skipped, 0 failures

## Scope (MVP)
- Core CRUD: create, list, view, edit, delete, batch-delete
- FK joins for user_name, site_name, card_name display
- remaining_amount = amount on create
- Card dropdown filtered by selected user

## Deferred
- Cashback auto-calculation from card rate
- Balance check display (expected vs actual)
- Consumed protection (immutable when FIFO-consumed)
- FIFO reprocessing triggers on edit/delete
- Active session warning
- Timestamp service integration

## Pitfalls / Follow-ups
- Amount stored as string via Decimal; currency display uses JS `Number().toFixed(2)` which may diverge from Python Decimal precision for edge cases
- Card filtering by user happens client-side; if user has many cards, may want server-side filter
- No consumed protection yet: editing a purchase that has been partially consumed by FIFO allocations could create accounting inconsistencies (deferred to follow-up)
- Activity nav group currently only has Purchases; Redemptions and Game Sessions will follow

## Test matrix
- Happy paths: create, list, update, delete, batch-delete, create with card, explicit SC received
- Edge cases: no time, manual cashback, empty workspace, pagination
- Failure injection: delete nonexistent, update nonexistent, batch-delete partial match, empty batch
- Model validation: zero amount, negative amount, missing user/site/date, defaults
