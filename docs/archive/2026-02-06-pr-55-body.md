Implements Issue #55 - Repair Mode for manual control of derived data rebuilds during troubleshooting.

## Changes

**Backend (`services/repair_mode_service.py`):**
- `RepairModeService` manages enabled state and stale pair tracking in settings.json
- `StalePair` dataclass with user_id, site_id, boundary date/time, timestamp, and reasons
- Methods: `is_enabled()`, `set_enabled()`, `mark_pair_stale()`, `clear_pair()`, `get_stale_pairs()`

**AppFacade Integration:**
- Added `_rebuild_or_mark_stale()` helper method (50+ lines)
- Conditionally performs immediate rebuild (normal mode) or marks pair stale (repair mode)
- Refactored 10+ CRUD methods to use helper:
  - `create_purchase()`, `update_purchase()`, `delete_purchase()`
  - `create_redemption()`, `update_redemption()`, `delete_redemption()`
  - Plus session, expense, and adjustment operations
- Cross-pair moves (e.g., purchase reassignment) mark both old and new pairs stale

**UI Implementation:**
- `ui/repair_mode_dialog.py`: Confirmation dialog with warning bullets and required acknowledgment checkbox
- `ui/tabs/tools_tab.py`: Repair Mode section in Tools tab
  - Status indicator (🔴 ENABLED / 🟢 Disabled)
  - Toggle button (red styling in normal mode to indicate danger)
  - Stale pairs count display
  - "Rebuild Stale Pairs" and "Clear Stale List" buttons
- `ui/main_window.py`:
  - Red banner at top when repair mode enabled
  - Window title suffix " - REPAIR MODE"
  - `refresh_repair_mode_ui()` method to update banner/title when toggled

**Settings:**
- `ui/settings.py`: Added defaults for `repair_mode_enabled` and `repair_mode_stale_pairs`

**Documentation:**
- `docs/status/CHANGELOG.md`: Entry 2026-02-06-05 with complete feature documentation
- `docs/PROJECT_SPEC.md`: New section 6.1 Repair Mode with architecture, workflow, and safety notes
- `docs/archive/2026-02-06-issue-55-proposed-body.md`: Issue #55 specification source

## Behavior

**Enable Repair Mode:**
1. User clicks "Enable Repair Mode" button in Tools tab
2. `RepairModeConfirmDialog` shows with explicit warnings
3. User must check acknowledgment checkbox to enable button
4. Blocked if Maintenance Mode is active (warning shown)
5. On confirmation: banner/title update, auto-rebuild disabled

**While Repair Mode Enabled:**
- All CRUD operations (purchase/redemption/session/adjustment create/update/delete) mark affected (user, site) pairs as stale
- Stale pairs persist in settings.json across app restarts
- UI shows stale pair count in Tools tab

**Rebuild/Clear:**
- "Rebuild Stale Pairs": Uses existing recalculation worker, clears stale markers after rebuild
- "Clear Stale List": Removes stale markers without rebuilding (for manually verified data)

**Disable Repair Mode:**
- Immediate (no confirmation)
- Banner and title removed
- Auto-rebuild resumes for future edits

- `refresh_repair_mode_ui()` rebuilds all tabs to update banner visibility (surgical DOM update would be more efficient)
- No visual indication in other tabs when repair mode is active (only banner at top)
- Stale pair list in Tools tab could show more details (boundary dates, reasons)

**Suggested Follow-ups (new Issues):**
