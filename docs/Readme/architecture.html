Architecture + data model (operator-focused)

App layers (why it matters)

Sezzions is structured as UI → services → repositories → SQLite. For operators, the key implication is: most multi-step edits are atomic. If an operation fails halfway through, it should roll back so you don’t get partial writes.

LayerRoleExamples
UI Forms, tables, dialogs, user workflows Main tabs, View/Edit dialogs, Tools UI
Services Business rules and orchestration FIFO rules, session P/L model, recalculation orchestration
Repositories SQLite reads/writes Purchases/Redemptions queries, bulk merge/reset
SQLite Single-file DB with transactional semantics sezzions.db, backups/restores

Database file

  • Default DB: sezzions.db in repo root
  • Override: set SEZZIONS_DB_PATH when starting the app

Authoritative vs derived data

When you see “wrong numbers”, it helps to know whether you’re looking at authoritative inputs (things you entered) or derived data (things Sezzions can rebuild).

Authoritative inputs (entered/edited)
  • Purchases
  • Redemptions
  • Game Sessions (start/end balances)
  • Expenses
  • Adjustments & checkpoints
Derived (rebuildable)
  • FIFO allocations
  • Realized transactions
  • Session recalculations
  • Event links

Maintenance Mode

If Sezzions detects data integrity issues at startup, it can enter Maintenance Mode. This reduces what parts of the app are available so you can repair/restore/recalculate without the UI crashing.

Typical maintenance steps: finish any pending CSV import, then run “Recalculate Everything”, or restore a known-good backup. See Maintenance Mode dialog.

Repair Mode (advanced)

Repair Mode disables automatic derived rebuilds after edits. Sezzions then tracks which (user, site) pairs are “stale”. You must explicitly rebuild stale pairs using Tools.

Rule of thumb: Don’t enable Repair Mode unless you have a reason (e.g., corruption triage or huge bulk edits).
